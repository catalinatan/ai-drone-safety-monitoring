# AI Safety Monitoring System

A real-time safety monitoring platform that uses computer vision to detect humans in restricted zones and automatically dispatches drones for investigation. Supports both AirSim simulation and real-world deployments with IP cameras, GPS-equipped hardware, and MAVLink-compatible drones.

## Overview

The system monitors multiple CCTV feeds for human intrusion into designated danger zones. When a person enters a restricted area, it calculates their 3D world coordinates using monocular depth estimation and camera projection, and can dispatch a drone to investigate autonomously.

**Key capabilities:**

- Real-time human detection using YOLO11 instance segmentation
- Three-tier danger zones (red / yellow / green) drawn via a web UI or auto-generated by scene segmentation models
- Monocular depth estimation (Lite-Mono) for 3D coordinate calculation
- Dual projection backends: AirSim simulation or config-based real-world projection
- PnP camera calibration tool for determining camera orientation from known reference points
- Live GPS position updates for moving platforms (e.g., ship-mounted cameras)
- Automatic drone deployment to intrusion locations
- Replay trigger system with human tracking masks across all replay frames
- Multi-feed MJPEG streaming at configurable FPS
- Admin panel for runtime configuration, camera pose editing, and calibration
- Swappable hardware backends: AirSim, RTSP/IP cameras, MAVLink drones, video files

## Quick Start

### Option A: AirSim Simulation

1. **Start AirSim** with your Unreal Engine environment (ensure the `settings.json` includes the required vehicles — see [AirSim Configuration](#airsim-configuration)).

2. **Launch all services:**

   ```bash
   python main.py
   ```

3. **Open the UI** at http://localhost:5173

The default `config/feeds.yaml` is pre-configured for AirSim with four CCTV camera drones (Drone2–Drone5) and one deployment drone (Drone1).

In AirSim mode, 3D coordinate projection is handled automatically — the system queries camera pose directly from the AirSim API. No manual camera pose configuration is needed.

#### Simulator mode (no fine-tuned model)

If you don't have the fine-tuned YOLO weights, use the stock model:

```bash
python main.py --simulator
```

This uses `yolo11n-seg` (downloaded automatically by ultralytics on first run).

### Option B: Real Hardware (IP Cameras)

For real-world deployments with physical CCTV cameras, the system uses config-based projection instead of AirSim. You provide the camera's position and orientation, and the system projects detected humans to 3D world coordinates using monocular depth estimation and ray-ground intersection.

#### Step 1: Configure camera feeds

Edit `config/feeds.yaml` to define your cameras:

```yaml
feeds:
  cam-north:
    name: "Bridge North Camera"
    location: "North Pillar"
    scene_type: "bridge"
    camera:
      type: "rtsp"
      params:
        url: "rtsp://192.168.1.100:554/stream1"
    position:
      latitude: 1.2847       # decimal degrees
      longitude: 103.8610    # decimal degrees
      altitude: 15.0         # metres above sea level
    orientation:
      pitch: -30.0     # degrees, negative = looking down
      yaw: 180.0       # degrees, compass heading (0=north, 90=east, 180=south)
      roll: 0.0        # degrees, usually 0
    fov: 90.0          # horizontal field of view in degrees

  cam-south:
    name: "Dock Camera"
    location: "Loading Bay"
    scene_type: "ship"
    camera:
      type: "rtsp"
      params:
        url: "rtsp://192.168.1.101:554/stream1"
    position:
      latitude: 1.2890
      longitude: 103.8625
      altitude: 10.0
    orientation:
      pitch: -25.0
      yaw: 0.0
      roll: 0.0
    fov: 70.0
```

#### Step 2: Determine camera position

You need to know each camera's GPS coordinates. Options:

- **GPS transponder**: If your camera has a GPS module, read the lat/lon/altitude directly.
- **Phone GPS**: Stand at the camera location and note the coordinates from a maps app.
- **Site survey / building plans**: Look up the coordinates from documentation.

Enter the position as decimal degrees (latitude, longitude) and altitude in metres above sea level. The system converts GPS to a local NED (North-East-Down) coordinate frame internally for the projection math.

#### Step 3: Determine camera orientation

You have three options, from easiest to hardest:

**Option 1 — PnP Calibration (recommended)**

Use the built-in calibration tool in the Admin Panel. This solves for your camera's orientation automatically:

1. Launch the system: `python main.py`
2. Open the UI at http://localhost:5173
3. Go to **Admin Panel** > find your camera in the **Camera Pose** section > click **Calibrate**
4. Click 4 or more points in the camera image that you can identify in the real world
5. For each point, enter its GPS coordinates (latitude, longitude, altitude)
6. Click **Calibrate** — the system uses OpenCV's PnP solver to compute pitch, yaw, and roll

Good calibration points: corners of buildings, lamp posts, bollards, road markings — anything with a known, fixed position.

**Option 2 — Manual entry in Admin Panel**

1. Go to **Admin Panel** > find your camera > click **Edit Pose**
2. Enter estimated pitch (how far it looks down), yaw (compass heading), and roll
3. Click **Save Pose**

**Option 3 — Set values directly in `feeds.yaml`**

Enter the `orientation` values as shown in the configuration example above.

#### Step 4: Launch the system

```bash
python main.py
```

Open the UI at http://localhost:5173 and draw danger zones on each camera feed (or use the auto-segment feature).

#### Moving platforms (ships, vehicles)

For cameras mounted on moving platforms, the system supports live GPS position updates via API. Your GPS transponder software should push position updates periodically:

```bash
curl -X POST http://localhost:8001/feeds/cam-north/position \
  -H "Content-Type: application/json" \
  -d '{"latitude": 1.2850, "longitude": 103.8615, "altitude": 15.0}'
```

You can also include a heading (yaw) update:

```bash

  -H "Content-Type: application/json" \
  -d '{"latitude": 1.2850, "longitude": 103.8615, "altitude": 15.0, "heading": 135.0}'
```

This updates the camera's projection in real time without restarting the system.

#### Adding a drone for real hardware

Configure the drone backend for MAVLink:


MAVLink connection string formats:
- Serial: `/dev/ttyUSB0` or `/dev/ttyUSB0:921600`
- UDP (SITL): `udp:127.0.0.1:14550`
- TCP: `tcp:127.0.0.1:5760`

## Projection System

The system uses a projection abstraction layer to convert detected humans from 2D pixel coordinates to 3D world coordinates. The projection backend is selected automatically based on the camera type.

### How it works

1. YOLO detects a person and produces a segmentation mask
2. The bottom-center of the mask is used as the ground contact point (feet position), rather than the centroid, to avoid vertical bias from torso-level sampling
3. Lite-Mono estimates relative depth at that pixel
4. The projection backend converts (pixel_x, pixel_y, depth) to (x, y, z) world coordinates using ray-ground plane intersection
5. Only the X and Y coordinates are used for drone dispatch — the Z coordinate is overridden by the configured `safe_altitude` for flight safety

### Projection backends

| Backend | Used when | Pose source |
|---------|-----------|-------------|
| `AirSimProjection` | Camera type is `airsim` | Queried live from AirSim API via `simGetCameraInfo()` |
| `ConfigProjection` | Camera type is `rtsp`, `file`, or any non-AirSim type | `feeds.yaml` config, Admin Panel, live GPS API, or PnP calibration |

The two backends are completely independent — changing configuration for real-world cameras has no effect on AirSim operation, and vice versa.

### ConfigProjection math

For real-world cameras, the system:

1. Computes camera intrinsics (focal length) from the configured horizontal FOV
2. Casts a ray from the camera through the detected pixel
3. Rotates the ray from camera frame to world frame using the configured orientation
4. Intersects the ray with the ground plane (z=0 in NED) to find the world position
5. Uses the monocular depth estimate to interpolate along the ray

### Camera height and ground plane

The projection accuracy depends critically on the camera height parameter (`cctv_height_meters`). This value defines the vertical distance between the camera and the ground plane. The system uses this to compute the ray-ground intersection:

```
ground_z = 0.0 (NED ground plane)
t_ground = (ground_z - cam_z) / ray_z
world_point = cam_position + t_ground * ray_direction
```

If the camera height is incorrect (e.g., the camera is mounted on a ship deck above sea level, but the configured height assumes ground level), projected coordinates will have systematic directional bias — all estimates will be shifted consistently in one direction.

**For AirSim cameras**, the system automatically calibrates the camera height on first use by querying the `ThirdPersonCharacter` actor's Z position and computing the actual camera-to-ground distance. This removes the need for manual height configuration in simulation.

**For real-world cameras**, the height can be determined in three ways:

1. **Direct measurement** — Measure the camera's height above the ground plane it observes
2. **Height calibration tool** — Use the Admin Panel's "Calibrate Height" feature (see below)
3. **Manual configuration** — Set `cctv_height_meters` in the camera's projection parameters

### Height calibration

The height calibration tool back-calculates the camera's height above ground from a single known ground point:

1. In the Admin Panel, select a camera and click **Calibrate Height**
2. Click a point on the ground in the camera snapshot — this must be a point whose real-world position you know
3. Enter the GPS coordinates (latitude, longitude) of that ground point
4. Click **Calibrate** — the system casts a ray through the clicked pixel and solves for the camera height `h` such that the ray-ground intersection lands at the specified world coordinates

This is particularly useful when the camera's exact height above the ground plane is ambiguous (e.g., cameras on bridges, ship decks, or elevated platforms where "height" could be measured relative to different surfaces).

### PnP calibration (orientation)

The PnP calibration tool uses OpenCV's `solvePnP` with the iterative method to determine camera orientation from 4+ point correspondences between pixel coordinates and known world coordinates. The solver outputs a rotation matrix which is decomposed into pitch, yaw, and roll angles.

Requirements for good calibration:
- At least 4 non-coplanar points (more points = better accuracy)
- Points should be spread across the camera's field of view
- World coordinates must be accurate GPS positions (latitude, longitude, altitude)

## How Detection Works

1. **Frame capture** — Background thread continuously grabs frames from all camera backends and stores them in the feed manager.
2. **YOLO inference** — A detection thread runs YOLO11 instance segmentation on each frame, producing per-person binary masks.
3. **Zone overlap** — Each person mask is tested against red and yellow zone masks. Overlap above the threshold triggers the corresponding alert level.
4. **Depth estimation** — For red zone intrusions, the bottom-center of the person's mask (feet) is used as the ground contact point. Lite-Mono estimates relative depth at that pixel.
5. **3D coordinate calculation** — The projection backend converts the 2D pixel + depth into 3D world coordinates (NED frame). For AirSim, this queries the simulator directly and auto-calibrates camera height from the scene. For real cameras, this uses ray-ground intersection with the configured (or calibrated) camera pose and height.
6. **Drone dispatch** — The first red zone intrusion triggers automatic drone deployment. Subsequent intrusions require manual confirmation via the UI.
7. **Replay capture** — The system continuously buffers recent frames with their detection masks. When an alarm triggers, the replay includes human tracking masks (cyan) for all buffered frames and the trigger mask (red) for the intrusion frame.

### Coordinate System (NED)

The system uses the NED (North-East-Down) coordinate frame:

| Axis | Direction | Sign convention |
|------|-----------|-----------------|
| **X** | North | Positive = forward |
| **Y** | East | Positive = right |
| **Z** | Down | Positive = below ground, **negative = above ground** |

## Depth Estimation Evaluation

The system includes an evaluation script (`src/eval/eval_depth_estimation.py`) for measuring depth estimation and projection accuracy against AirSim ground truth. It compares the system's estimated world coordinates with the actual position of a `ThirdPersonCharacter` actor in the AirSim environment.

### Running the evaluation

```bash
# Basic evaluation on a single environment
python -m src.eval.eval_depth_estimation --env ship

# With live camera visualization
python -m src.eval.eval_depth_estimation --env ship --show

# Setup drones in follow mode before collecting samples
python -m src.eval.eval_depth_estimation --env ship --setup

# Aggregate results across multiple environments
python -m src.eval.eval_depth_estimation --aggregate
```

### Command-line flags

| Flag | Description |
|------|-------------|
| `--env <name>` | AirSim environment to evaluate (e.g., `ship`, `bridge`) |
| `--show` | Display per-camera OpenCV windows with live detection overlay |
| `--setup` | Start the drone follow-mode loop (mirrors production `--follow` behaviour) before evaluation |
| `--aggregate` | Combine results from multiple `--env` runs into a single summary |

### How it works

1. **Follow mode** — When `--setup` is used, CCTV drones replicate the production follow-mode loop: each drone is continuously teleported to its corresponding Camera Actor pose in the AirSim scene.
2. **Detection** — YOLO runs on each camera frame. The bottom of each detected person mask (feet position) is used as the ground contact point.
3. **Filtering** — Small detections below `MIN_MASK_AREA_FRACTION` (0.1% of frame area) are discarded to eliminate false positives from background objects.
4. **Matching** — Each detection is matched to the `ThirdPersonCharacter` ground truth position. Matches exceeding `MATCH_DISTANCE_THRESHOLD_M` are discarded to avoid false matches when the character isn't visible.
5. **Projection** — The matched pixel is projected to world coordinates using the same `AirSimProjection` backend as production, with the camera height auto-calibrated from AirSim poses.
6. **Error calculation** — Only X and Y errors are measured (2D horizontal plane), since the Z coordinate is overridden by `safe_altitude` for drone dispatch and is not operationally relevant.

### Output

Results are saved to `eval_output/depth_estimation/<env>/`:

| File | Description |
|------|-------------|
| `results.csv` | Per-sample raw data (pixel coords, estimated/ground-truth X/Y, errors, distance) |
| `summary.json` | Aggregate statistics (mean, median, P90, max errors) |
| `error_histogram.png` | Distribution of 2D position errors |
| `scatter_xy.png` | Scatter plot of estimated vs ground truth positions with error vectors |
| `error_vs_distance.png` | Error magnitude as a function of camera-to-person distance |

## Troubleshooting

### Backend starts but feeds show "NO SIGNAL"

- **AirSim**: Ensure the simulator is running before starting the backend. Verify vehicle names in `config/feeds.yaml` match your `settings.json`.
- **RTSP**: Confirm the camera URL is reachable (`ffprobe rtsp://...`). Check firewall rules.
- **File**: Verify the video file path exists and is readable.

### Low detection frame rate

- Check GPU utilization: `nvidia-smi`
- Verify CUDA is available: `python -c "import torch; print(torch.cuda.is_available())"`
- Lower `detection.fps` or `detection.inference_imgsz` in the Admin panel or `config/default.yaml`

### Zones not persisting

Zones are saved to `data/zones.json`. Ensure the backend process has write permissions to the `data/` directory.

### Drone not responding

1. Check the drone server is running: `curl http://localhost:8000/status`
2. Ensure the drone is in **automatic** mode for navigation commands
3. Verify target coordinates are within safety limits
4. For AirSim: check the simulator console for errors
5. For MAVLink: verify the serial connection and that the flight controller is powered

### Auto-segmentation returns no zones

- Ensure the scene segmentation model weights exist for the selected scene type
- Try lowering `auto_segmentation.confidence` in the Admin panel
- Check backend logs for model loading errors

### Camera pose / projection issues

- Verify position values are in metres and use NED convention (negative Z = above ground)
- If using PnP calibration, ensure points are spread across the image and world coordinates are accurate
- Check the Admin Panel > Camera Pose section to verify the current values
- For AirSim cameras, the Camera Pose section has no effect — pose is read from the simulator
- **Systematic directional bias in projections** — If all projected coordinates are consistently shifted in one direction, the camera height is likely incorrect. Use the height calibration tool (Admin Panel > Calibrate Height) to recalculate it from a known ground point
- **Height ambiguity on elevated platforms** — For cameras on bridges, ship decks, or rooftops, the "height" should be measured relative to the ground plane the camera observes (e.g., the deck surface), not relative to sea level or the building base
