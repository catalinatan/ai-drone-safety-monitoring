# AI Safety Monitoring System

A real-time safety monitoring platform that uses computer vision to detect humans in restricted zones and automatically dispatches drones for investigation. Designed with a hardware abstraction layer that supports both AirSim simulation and real-world deployments with IP cameras and MAVLink-compatible drones.

## Overview

The system monitors multiple CCTV feeds for human intrusion into designated danger zones. When a person enters a restricted area, it calculates their 3D world coordinates using monocular depth estimation and can dispatch a drone to investigate autonomously.

**Key capabilities:**

- Real-time human detection using YOLO11 instance segmentation
- Three-tier danger zones (red / yellow / green) drawn via a web UI or auto-generated by scene segmentation models
- Monocular depth estimation (Lite-Mono) for 3D coordinate calculation
- Automatic drone deployment to intrusion locations
- Multi-feed MJPEG streaming at configurable FPS
- Admin panel for runtime configuration without restarts
- Swappable hardware backends: AirSim, RTSP/IP cameras, MAVLink drones, video files

## Architecture

```
┌──────────────┐       ┌──────────────────┐       ┌──────────────┐
│  React UI    │◄─────►│  FastAPI Backend  │◄─────►│ Drone Server │
│  (Vite)      │       │  (Port 8001)     │       │ (Port 8000)  │
│  Port 5173   │       │                  │       │              │
└──────────────┘       └────────┬─────────┘       └──────┬───────┘
                                │                        │
                    ┌───────────┴───────────┐            │
                    ▼                       ▼            ▼
          ┌──────────────────┐    ┌─────────────────────────────┐
          │ Hardware Layer   │    │   Drone Backend             │
          │ (Camera backends │    │   (AirSim or MAVLink)       │
          │  per feed)       │    │                             │
          └──────────────────┘    └─────────────────────────────┘
```

The system consists of three services:

| Service | Port | Purpose |
|---------|------|---------|
| **Backend API** | 8001 | Detection pipeline, feed management, zone management, MJPEG streaming |
| **Drone Server** | 8000 | Drone control API (goto, return home, mode switching, camera streams) |
| **React UI** | 5173 | Command center dashboard, zone editor, drone control panel, admin settings |

### Module Organization

| Layer | Path | Description |
|-------|------|-------------|
| **Core** | `src/core/` | Pure business logic — no I/O, no framework dependencies. Zone overlap detection, alarm state machine, detection pipeline orchestration. |
| **Hardware** | `src/hardware/` | Swappable camera and drone backends via abstract base classes and factory functions. |
| **Services** | `src/services/` | Stateful business logic — feed state management, zone persistence, MJPEG encoding, drone dispatch policies. |
| **Detection** | `src/detection/` | ML model wrappers — YOLO human detector, scene segmenter, depth estimator. |
| **Spatial** | `src/spatial/` | 3D coordinate utilities — ray-ground plane intersection, camera projection matrices. |
| **API** | `src/api/` | FastAPI routes and background loops (frame capture, detection). |
| **Drone Server** | `src/drone_server/` | Separate FastAPI app for drone control with its own background control loop. |
| **UI** | `src/ui/` | React + TypeScript + Tailwind CSS frontend. |

```bash
git clone <repository-url>

# Core install (editable mode)
pip install -e .

# With development tools (pytest, ruff, httpx)
pip install -e ".[dev]"
```

### 2. Install PyTorch with CUDA

PyTorch must be installed separately to match your CUDA version:

```bash
# CUDA 12.4 (recommended)
pip install torch torchvision --index-url https://download.pytorch.org/whl/cu124

# CUDA 12.1
pip install torch torchvision --index-url https://download.pytorch.org/whl/cu121

# CPU only (not recommended for real-time use)
pip install torch torchvision --index-url https://download.pytorch.org/whl/cpu
```

### 3. Install hardware-specific extras

```bash
# For AirSim simulation
pip install -e ".[airsim]"

# For real MAVLink drones (PX4, ArduPilot)
pip install -e ".[mavlink]"
```

### 4. Install frontend dependencies

```bash
cd src/ui
npm install
cd ../..
```

### 5. Model weights

| Model | Location | Notes |
|-------|----------|-------|
| YOLO11 human detection | `models/human_detection/yolo11s-seg/weights/best.pt` | Fine-tuned weights. For simulation testing, use `--simulator` flag to fall back to stock `yolo11n-seg`. |
| Scene segmentation (bridge) | `models/scene_segmentation/bridge/bridge_hazard_yolo11s-seg/weights/best.pt` | Auto-generates danger zones for bridge scenes. |
| Scene segmentation (railway) | `models/scene_segmentation/railway/railway_hazard_yolo11s-seg/weights/best.pt` | Auto-generates danger zones for railway scenes. |
| Scene segmentation (ship) | `models/scene_segmentation/ship/ship_hazard_yolo11s-seg/weights/best.pt` | Auto-generates danger zones for ship/maritime scenes. |
| Lite-Mono depth estimation | `models/depth_estimation/lite-mono-small_640x192/encoder.pth` and `depth.pth` | Monocular depth for 3D coordinate calculation. |

## Quick Start

### Option A: AirSim Simulation

1. **Start AirSim** with your Unreal Engine environment (ensure the `settings.json` includes the required vehicles — see [AirSim Configuration](#airsim-configuration)).

2. **Launch all services:**

   ```bash
   python main.py
   ```

3. **Open the UI** at http://localhost:5173

The default `config/feeds.yaml` is pre-configured for AirSim with four CCTV camera drones (Drone2–Drone5) and one deployment drone (Drone1).

#### Simulator mode (no fine-tuned model)

If you don't have the fine-tuned YOLO weights, use the stock model:

```bash
python main.py --simulator
```

This uses `yolo11n-seg` (downloaded automatically by ultralytics on first run).

### Option B: Real Hardware (IP Cameras + MAVLink Drone)

1. **Configure your camera feeds** in `config/feeds.yaml`:

   ```yaml
   feeds:
     cam-1:
       name: "Entrance Camera"
       location: "North Gate"
       scene_type: "bridge"
       camera:
         type: "rtsp"
         params:
           url: "rtsp://192.168.1.100:554/stream1"

     cam-2:
       name: "Dock Camera"
       location: "Loading Bay"
       scene_type: "ship"
       camera:
         type: "rtsp"
         params:
           url: "rtsp://192.168.1.101:554/stream1"
   ```

2. **Configure the drone backend.** Edit `src/drone_server/config.yaml` to match your setup, and set the drone type in your environment or configuration. The drone server uses the `create_drone_backend()` factory — configure it for MAVLink:

   ```yaml
   # In your drone configuration
   drone:
     type: "mavlink"
     params:
       connection_string: "/dev/ttyUSB0"  # Serial port to flight controller
   ```

   MAVLink connection string formats:
   - Serial: `/dev/ttyUSB0` or `/dev/ttyUSB0:921600`
   - UDP (SITL): `udp:127.0.0.1:14550`
   - TCP: `tcp:127.0.0.1:5760`

3. **Launch the system:**

   ```bash
   python main.py
   ```

4. **Open the UI** at http://localhost:5173 and draw danger zones on each camera feed (or use the auto-segment feature).

### Option C: Demo with Video Files

Use pre-recorded video files for testing without any hardware:

```yaml
# config/feeds.yaml
feeds:
  demo-1:
    name: "Demo Feed"
    location: "Test"
    scene_type: "bridge"
    camera:
      type: "file"
      params:
        path: "data/videos/demo.mp4"
```

## Configuration

### Global Settings — `config/default.yaml`

Runtime settings for the entire system. All values can be changed live via the Admin panel in the UI (changes apply immediately but do not persist to disk across restarts).

```yaml
server:
  backend_port: 8001
  drone_api_port: 8000
  frontend_port: 5173

detection:
  model_path: "models/human_detection/yolo11s-seg/weights/best.pt"
  confidence_threshold: 0.25
  inference_imgsz: 1280
  fps: 30
  warmup_frames: 20
  min_person_area_pixels: 2

zones:
  overlap_threshold: 0.5
  alarm_cooldown_seconds: 5.0
  persistence_file: "data/zones.json"

auto_segmentation:
  scene_type: "bridge"           # "bridge" | "railway" | "ship"
  enabled: false                 # auto-refresh zones on interval
  interval_seconds: 60.0
  confidence: 0.5
  simplify_epsilon: 2.0
  min_contour_area: 40.0

streaming:
  capture_fps: 30
  stream_fps: 30

drone:
  api_url: "http://localhost:8000"
  api_timeout: 5
  safe_altitude: -10.0
```

### Feed Configuration — `config/feeds.yaml`

Defines each camera feed and its hardware backend. The camera `type` field selects the backend:

| Type | Description | Required params |
|------|-------------|-----------------|
| `airsim` | AirSim simulator camera | `camera_name`, `vehicle_name` |
| `rtsp` | IP camera via RTSP/HTTP | `url` |
| `file` | Video file playback | `path` |

```yaml
feeds:
  my-camera:
    name: "My Camera"
    location: "Building A"
    scene_type: "bridge"        # Used for auto-segmentation model selection
    camera:
      type: "rtsp"
      params:
        url: "rtsp://192.168.1.100:554/stream"
```

### Environment Variable Overrides

Any key in `config/default.yaml` can be overridden via environment variables using `UPPER_SNAKE_CASE`:

```bash
DETECTION_FPS=15 python main.py
```

## Using the Web UI

### Command Center

The main dashboard shows all camera feeds in a configurable grid layout. Each feed displays:

- Live MJPEG stream with detection overlay
- People count and zone intrusion alerts
- Per-feed detection toggle (enable/disable)

**Header controls:**
- **Auto Segment** — Run scene segmentation on all feeds to generate danger zones automatically
- **Zones** — Toggle zone overlay visibility
- **Grid selector** — Choose how many feeds to display (1, 2, 4, 6, 9, or all)
- **Admin** — Open the system configuration panel

### Zone Editor

Click the edit button on any feed card to open the zone editor:

1. Draw polygons on the camera frame to define danger zones
2. Assign a zone level: **Red** (alarm + drone dispatch), **Yellow** (caution alert), or **Green** (safe)
3. Save zones — they persist across restarts via `data/zones.json`

Zones use percentage-based coordinates (0–100) so they remain valid at any resolution.

### Admin Panel

Runtime system configuration. Changes take effect immediately (no restart needed) but do not persist to the YAML file.

Configurable sections:
- **Detection** — Model confidence, inference size, FPS, warmup frames
- **Zones** — Overlap threshold, alarm cooldown
- **Auto-Segmentation** — Scene type (bridge/railway/ship), auto-refresh toggle and interval, segmentation confidence and polygon simplification
- **Streaming** — Capture and stream FPS
- **Drone** — API URL, timeout, safe altitude
- **Equipment** — Drone payload settings (e.g., lifevest deployment)

### Drone Control Panel

A collapsible side panel for monitoring and controlling the deployment drone:

- Real-time drone status (position, mode, navigation state)
- Forward and downward camera feeds
- Manual mode controls and automatic deployment triggers
- Trigger history showing past intrusion events

## How Detection Works

1. **Frame capture** — Background thread continuously grabs frames from all camera backends and stores them in the feed manager.
2. **YOLO inference** — A detection thread runs YOLO11 instance segmentation on each frame, producing per-person binary masks.
3. **Zone overlap** — Each person mask is tested against red and yellow zone masks. Overlap above the threshold triggers the corresponding alert level.
4. **Depth estimation** — For red zone intrusions, the person's feet position (bottom of mask) is located. Lite-Mono estimates depth at that pixel.
5. **3D coordinate calculation** — Ray-ground plane intersection converts the 2D pixel + depth into 3D world coordinates (NED frame).
6. **Drone dispatch** — The first red zone intrusion triggers automatic drone deployment. Subsequent intrusions require manual confirmation via the UI.

### Coordinate System (NED)

The system uses the NED (North-East-Down) coordinate frame:

| Axis | Direction | Sign convention |
|------|-----------|-----------------|
| **X** | North | Positive = forward |
| **Y** | East | Positive = right |
| **Z** | Down | Positive = below ground, **negative = above ground** |

## API Reference

### Backend API — Port 8001

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/health` | GET | System health check |
| `/feeds` | GET | List all feeds with detection status |
| `/feeds/{feed_id}/status` | GET | Detection status for a specific feed |
| `/feeds/{feed_id}/zones` | POST | Update danger zones |
| `/feeds/{feed_id}/auto-segment` | POST | Auto-generate zones via scene segmentation |
| `/feeds/{feed_id}/detection` | PATCH | Enable/disable detection for a feed |
| `/video_feed/{feed_id}` | GET | MJPEG video stream |
| `/triggers` | GET | List intrusion trigger history |
| `/config` | GET | Current runtime configuration |
| `/config` | PUT | Update runtime configuration |

Interactive API docs: http://localhost:8001/docs

### Drone Server — Port 8000

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/status` | GET | Drone status and position |
| `/mode` | POST | Set mode (`manual` / `automatic`) |
| `/goto` | POST | Navigate to NED coordinates |
| `/return_home` | POST | Return to takeoff position |
| `/video_feed/down` | GET | Downward camera MJPEG stream |
| `/video_feed/forward` | GET | Forward camera MJPEG stream |

Interactive API docs: http://localhost:8000/docs

### Zone Format

```json
{
  "zones": [
    {
      "id": "zone-1",
      "level": "red",
      "source": "manual",
      "points": [
        { "x": 10, "y": 10 },
        { "x": 50, "y": 10 },
        { "x": 50, "y": 50 },
        { "x": 10, "y": 50 }
      ]
    }
  ]
}
```

Zone levels:
- **Red** — Triggers alarm and automatic drone deployment on first intrusion
- **Yellow** — Caution alert (visual warning only, no drone)
- **Green** — Safe zone (no action)

## Troubleshooting

### Backend starts but feeds show "NO SIGNAL"

- **AirSim**: Ensure the simulator is running before starting the backend. Verify vehicle names in `config/feeds.yaml` match your `settings.json`.
- **RTSP**: Confirm the camera URL is reachable (`ffprobe rtsp://...`). Check firewall rules.
- **File**: Verify the video file path exists and is readable.

### Low detection frame rate

- Check GPU utilization: `nvidia-smi`
- Verify CUDA is available: `python -c "import torch; print(torch.cuda.is_available())"`
- Lower `detection.fps` or `detection.inference_imgsz` in the Admin panel or `config/default.yaml`

### Zones not persisting

Zones are saved to `data/zones.json`. Ensure the backend process has write permissions to the `data/` directory.

### Drone not responding

1. Check the drone server is running: `curl http://localhost:8000/status`
2. Ensure the drone is in **automatic** mode for navigation commands
3. Verify target coordinates are within safety limits
4. For AirSim: check the simulator console for errors
5. For MAVLink: verify the serial connection and that the flight controller is powered

### Auto-segmentation returns no zones

- Ensure the scene segmentation model weights exist for the selected scene type
- Try lowering `auto_segmentation.confidence` in the Admin panel
- Check backend logs for model loading errors

## Development

# Lint
ruff check src/

### Adding a new camera feed

1. Add an entry to `config/feeds.yaml` with the camera backend configuration
2. Restart the backend — the feed will appear in the UI automatically

### Adding a new camera backend

1. Create a new class in `src/hardware/camera/` implementing `CameraBackend`
2. Register it in `src/hardware/__init__.py` → `create_camera_backend()`

### Adding a new drone backend

1. Create a new class in `src/hardware/drone/` implementing `DroneBackend`
2. Register it in `src/hardware/__init__.py` → `create_drone_backend()`
